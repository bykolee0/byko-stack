# 스펙 문서 템플릿

## A. 단일 스펙

```markdown
# [프로젝트명] 스펙

## 목표
[이 프로젝트가 달성하려는 핵심 목표. 1-3문장으로 명확하게.]

## 배경
[왜 이 작업이 필요한지. 비즈니스 맥락, 기술적 동기, 관련 의사결정.]

## 참고자료
[외부 API 문서, 디자인 시안, 관련 이슈 링크 등]
- 외부와는 구성과 체계가 다를 수 있으므로, 참고만 하고 우리 시스템 표준을 정립한다

## 현행 분석 요약
[코드베이스 분석 핵심 결과. 상세는 analysis/ 아래 문서 또는 본 섹션의 파일:라인 근거 참조]
- 관련 코드 위치 및 패턴
- 적용할 컨벤션
- 주의사항

## 도메인 규칙
[해당 도메인의 비즈니스 규칙, 제약조건, 특수한 로직]

## 구현 상세
[API 명세, 데이터 모델, 로직 흐름 등 구체적 구현 내용]

### [기능 1]
- 설명
- 입력/출력
- 처리 로직
- 예외 케이스

### [기능 2]
...

## Acceptance Criteria

| ID | Acceptance Criteria | 검증 방법 |
|----|---------------------|----------|
| AC-1 | [구체적이고 검증 가능한 기준] | [테스트 유형 / 수동 확인 등] |
| AC-2 | ... | ... |

## 테스트 전략
- 테스트 범위: 어떤 기능/케이스를 테스트할 것인지
- 테스트 수준: unit / integration / e2e 중 어떤 수준이 적절한지
- 기존 패턴이 있다면 어떤 패턴을 따를 것인지
- 기존 패턴이 없다면 어떤 방식을 제안하는지와 그 이유

## 이슈 및 결정 사항
- [결정됨] ...
- [TBD] ...

## 구현 순서
1. ...
2. ...

## 구현 지침
- 기존 코드베이스 컨벤션을 분석하고 따를 것
- 모호한 내용은 먼저 조사하고, 유저 결정 조건에 해당할 때만 선택지와 추천을 제시할 것
- 스펙에 없는 기능을 추가하지 말 것
```

## B. 멀티 스펙

범위가 커서 분할이 필요한 경우, 메인 스펙 + 서브 스펙 구조를 사용한다.

### 메인 스펙 (`spec.md`)

```markdown
# [프로젝트명] 메인 스펙

## 목표
[전체 프로젝트가 달성하려는 핵심 목표]

## 배경
[왜 이 작업이 필요한지]

## 참고자료
[전체 프로젝트에 공통으로 적용되는 참고자료]

## 현행 분석 요약
[코드베이스 분석 핵심 결과. 상세는 analysis/ 아래 문서 또는 본 섹션의 파일:라인 근거 참조]

## 프로젝트 구조
[서브 스펙 간의 관계, 의존성, 전체 아키텍처 개요]

## 공통 규칙
[모든 서브 스펙에 공통으로 적용되는 도메인 규칙, 컨벤션, 제약조건]

## 서브 스펙 진행 현황
| 순서 | 서브 스펙 | 설명 | 상태 | 의존성 |
|------|-----------|------|------|--------|
| 1 | [01-feature-a.md](sub-specs/01-feature-a.md) | 기능 A 설명 | [ ] 미착수 | - |
| 2 | [02-feature-b.md](sub-specs/02-feature-b.md) | 기능 B 설명 | [ ] 미착수 | 01 완료 후 |

상태: [ ] 미착수 → [~] 스펙 작성중 → [S] 스펙 완료 → [D] 구현 완료

## 구현 지침
[전체 프로젝트에 공통으로 적용되는 구현 지침]
```

### 서브 스펙 (`sub-specs/01-feature-a.md`)

단일 스펙 템플릿(A)과 동일한 구조를 따르되, 상단에 메인 스펙 참조를 추가:

```markdown
# [서브 프로젝트명] 스펙

> 메인 스펙: [../spec.md](../spec.md)
> 공통 규칙 및 컨벤션은 메인 스펙을 참조한다.

(이하 단일 스펙 템플릿과 동일)
```

서브 스펙은 각각 독립적으로 Phase 2~5를 거친다.
서브 스펙이 완료될 때마다 메인 스펙의 진행 현황 테이블을 업데이트한다.

## 작성 원칙

- 한국어로 작성하되, 코드/기술 용어는 영어를 유지한다
- 구현자(사람 또는 에이전트)가 읽고 바로 작업에 착수할 수 있을 만큼 구체적으로 작성한다
- 코드/문서로 확인 가능한 것은 조사해 근거를 남긴다. 유저 결정 조건에 해당하는 것만 선택지와 추천을 제시해 묻고, 나머지는 `assumed`로 기록한다
- 유저와 대화 중 결정된 사항은 즉시 스펙에 반영한다

## 중단 후 재개

manifest.md, spec.md, ambiguity-ledger.md가 주어지면:

1. manifest에서 문제 정의, 산출물, 단계 상태를 확인한다
2. ledger에서 현재 게이트 상태를 확인한다
3. 스펙의 진행 현황을 파악한다
4. 유저에게 현재 상황을 요약하고 다음 단계를 제안한다
