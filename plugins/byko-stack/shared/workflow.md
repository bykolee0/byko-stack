# byko-stack 워크플로우 컨벤션

byko-stack의 모든 스킬이 공유하는 운영 원칙이다. 각 스킬은 트리거 시 이 문서를 읽고 따른다.

## 핵심 구조: 메인 세션 = 오케스트레이터

메인 세션의 역할은 **대화, 결정, 조율, 컨텍스트 주입**이다. 대량 파일 읽기·코드 분석·독립 평가·병렬 구현은 서브에이전트에 위임한다. 이유: 메인 컨텍스트에 분석 원문이 쌓이면 (1) 컨텍스트가 고갈되고 (2) 자기 산출물에 대한 확증 편향이 생긴다.

| 작업 | 수행 주체 |
|------|----------|
| 유저와의 문답, 결정 사항 정리 | 메인 세션 |
| 소규모 확인 (파일 1-2개, 빠른 Grep) | 메인 세션 직접 |
| 광범위 코드 분석 | `byko-stack:code-explorer` |
| 정합성 평가 (스펙/계획/구현) | `byko-stack:evaluator` |
| fresh-eyes 리뷰 | `byko-stack:reviewer` |
| 독립적 구현 단위 | `byko-stack:implementer` |

**서브에이전트 운용 규칙:**

1. **파일로 주고받는다.** 서브에이전트의 산출물은 작업 디렉토리의 파일로 저장한다 (`analysis/`, `eval-results/`, `review-results/`). 메인 세션은 결론·요약만 컨텍스트에 유지하고, 상세는 파일 경로로 참조한다.
2. **주입은 최소·중립으로.** 서브에이전트에는 문제 정의, 핵심 결정(매니페스트에서 발췌), 대상 경로, 산출물 저장 경로만 전달한다. evaluator/reviewer에게는 **현재 세션의 결론이나 기대를 누설하지 않는다** — 독립성이 평가의 가치다.
3. **독립적인 작업은 병렬로** 같은 턴에 띄운다.
4. **결과를 그대로 믿지 않는다.** 서브에이전트의 FAIL/지적 사항은 메인 세션이 근거를 확인한 후 채택한다.

## 워크 매니페스트

작업 단위마다 `manifest.md`가 산출물 경로와 상태의 단일 진실 공급원이다. 파일명·위치 하드코딩 대신 매니페스트가 인터페이스가 되므로, 어떤 스킬이든 독립적으로 호출되어 이어받을 수 있다.

**기본 위치:** `docs/specs/<project-name>/manifest.md` (산출물들과 같은 디렉토리)

**탐색 규칙 (모든 스킬 공통):**

1. 유저가 경로를 인자로 주면 그것을 쓴다 (매니페스트든 개별 문서든)
2. 없으면 `docs/specs/*/manifest.md`를 탐색한다 — 1개면 사용, 복수면 최근 수정순으로 제시하고 유저가 선택
3. 매니페스트가 없으면: 산출물을 만드는 스킬(spec-designer, spec-dev)은 새로 생성한다. 평가만 하는 스킬(eval-gate, review)은 인자 기반으로 동작하고, 매니페스트가 있을 때만 갱신한다

**템플릿:**

```markdown
# Work Manifest: <project-name>

> 작업 디렉토리: docs/specs/<project-name>/
> 마지막 업데이트: YYYY-MM-DD HH:MM (<skill-name>)

## 문제 정의
<무엇을 왜 만드는가, 2-5줄. reviewer가 출발점으로 삼는 원본 문제 서술 —
스펙이 진화해도 이 섹션은 최초 요구의 본질을 유지한다>

## 산출물
| 산출물 | 경로 | 상태 |
|--------|------|------|
| 스펙 | spec.md | ✅ |
| ambiguity ledger | ambiguity-ledger.md | ✅ 게이트 통과 |
| 코드 분석 | analysis/<topic>.md | ✅ |
| 구현 계획 | implementation-plan.md | — |
| traceability | traceability.md | — |
| eval 결과 | eval-results/ | spec: APPROVED |
| review 결과 | review-results/ | — |

## 단계 상태
- [x] 스펙 설계 — 완료
- [x] 스펙 검증 (eval-gate) — APPROVED
- [ ] 구현 — AC 3/7
- [ ] 구현 검증 (eval-gate)
- [ ] 리뷰 (review)

## 핵심 결정
<서브에이전트 주입용. 결정 + 근거를 한 줄씩>

## 미해결
<open/assumed 항목, [TBD] 목록>
```

산출물 경로는 매니페스트 기준 상대 경로. 단계는 고정 순서가 아니다 — 실제로 수행한 단계만 기록하고, 건너뛴 단계는 적지 않는다.

**갱신 책임:** 산출물을 만들거나 상태를 바꾼 스킬이 종료 전에 매니페스트를 갱신한다.

## 핸드오프

각 스킬은 작업 완료 시 매니페스트 상태를 근거로 자연스러운 다음 단계를 1-2개 제안한다. 강제하지 않는다 — 유저는 어떤 스킬이든 어떤 순서로든 호출할 수 있고, 모든 스킬은 매니페스트만으로 맥락을 복원할 수 있어야 한다.

| 방금 끝난 일 | 자연스러운 다음 제안 |
|--------------|---------------------|
| 스펙 완성 | `/eval-gate spec` 또는 바로 `/spec-dev` |
| 계획 수립 | `/eval-gate plan` 또는 바로 구현 |
| 구현 완료 | `/eval-gate implementation`, `/review` |
| eval NEEDS_REVISION | 수정 담당 스킬로 복귀 |
| review CONCERNS 이상 | 해당 산출물 스킬로 복귀 |

## 질문 정책

`shared/question-policy.md`를 따른다. 요약: **조사 → 해결안 작성 → 유저 결정이 필요한 경우에만 선택 요청.** 코드로 답할 수 있는 것을 유저에게 묻지 않는다.
