# 질문 정책: 조사 → 제안 → 선택

## 왜 이 정책인가

두 가지 실패 모드를 동시에 막는다:

- **임의 개발**: 모호한 항목을 AI가 조용히 자체 결정 → 유저가 원치 않는 방향으로 진행
- **과잉 질문**: 코드를 읽으면 답이 나오는 것까지 유저에게 질문 → 스킬을 쓰는 것이 오히려 더 많은 사람 손을 요구

원칙: **유저의 시간은 "정말 유저만 내릴 수 있는 결정"에만 쓴다.** 나머지는 AI가 조사하고, 결정하고, 근거와 함께 기록한다.

## 절차

모호한 항목을 만나면:

### 1. 먼저 조사한다

- 코드베이스 (직접 Grep/Read 또는 code-explorer 위임)
- 기존 산출물 (스펙, 매니페스트, 이전 결정 사항)
- 내부 문서 (`docs/`), 언급된 외부 자료

### 2. 조사로 답이 특정되면 — 질문하지 않는다

채택하고 ledger에 `from-code`로 기록한다 (파일:라인 근거 필수). 유저에게는 다음 응답에서 "확인한 사실" 목록으로 보고만 한다 — 답변을 기다리는 질문으로 만들지 않는다.

예: "현재 인증 방식은?", "테스트 프레임워크는?", "기존 에러 포맷은?" → 전부 코드에 답이 있다. 묻지 않는다.

### 3. 답이 하나로 특정되지 않으면 — 해결안을 작성한다

해결안 2-3개를 만든다. 각각에:

- 코드 근거 (기존 패턴과의 정합성)
- 트레이드오프
- 추천 여부와 이유

### 4. 유저 결정 조건에 해당하는가?

다음 중 하나라도 해당하면 **선택지 + 추천과 함께** 유저에게 묻는다:

| 조건 | 예시 |
|------|------|
| 고객/제품 요구사항에 영향 | 기능 범위, 대상 사용자, 허용 동작 |
| 비즈니스 정책 | soft/hard delete, 재시도 정책, 플랜별 제공 여부 |
| 기존 컨벤션과 충돌 또는 새 컨벤션 정립 | 새 에러 포맷 도입, 디렉토리 구조 변경 |
| 프로젝트 방향성 | 아키텍처 선택, 의존성 추가 |
| 되돌리기 어려운 사이드이펙트 | DB 스키마, 공개 API, 데이터 마이그레이션 |
| 비용/보안에 실질적 영향 | 외부 서비스 과금, 권한 모델 |

### 5. 해당하지 않으면 — 추천안을 채택하고 진행한다

ledger에 `assumed`로 기록한다: 채택안 + 근거 + 검토했던 대안. 산출물 제시 시점에 "자체 결정한 사항" 섹션으로 모아 보여준다 — 유저는 한 번에 훑고 이의만 제기하면 된다.

`assumed` 게이트(≤2)는 유지된다. 자체 결정이 누적되면 게이트에 걸리므로, 과도한 임의 진행은 구조적으로 차단된다. 게이트에 걸리면 assumed 항목들을 모아 일괄 확인을 요청한다.

## 질문의 형태

묻기로 했다면:

- **열린 질문 금지.** "어떻게 처리할까요?" (X) → 조사 기반 선택지 2-3개 + 추천 (O)
- 한 번에 3개 이하. 각 선택지에 트레이드오프 1줄
- 유저가 "나중에"라고 하면 `[TBD]`로 기록하고 진행 — 구현 시점에 다시 묻는다

## 자기 점검

- 이 질문의 답이 코드/문서 어딘가에 있는가? → 있으면 묻지 말고 찾는다
- 해결안 없이 열린 질문을 던지려 하는가? → 먼저 조사해서 선택지를 만든다
- 유저 결정 조건 표에 해당하지 않는데 묻고 있는가? → 추천안 채택 + assumed 기록으로 전환
- 유저 결정 조건에 해당하는데 조용히 진행하려 하는가? → 멈추고 묻는다
