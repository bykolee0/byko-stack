# 모드별 평가 체크리스트

## 공통 원칙

### 코드베이스 정합성 — 능동적으로 검증한다

스펙/계획/구현은 모두 코드베이스 위에 성립한다. 문서가 코드에 대한 주장을 포함하지 않더라도 평가자는 대상 영역을 직접 스캔하여 코드베이스 상태와의 정합성을 확인한다. 문서만 보고 판단하지 않는다.

문서의 주장은 검증한다:
- "N곳에서 사용된다" → Grep으로 실제 사용처 수 확인
- "기존 패턴은 X다" → 실제 코드를 읽어 확인
- "영향 범위는 A, B다" → 호출 체인을 추적하여 누락 확인

주장이 없어도 능동적으로 찾는다:
- 대상 영역 주변 모듈/호출 체인을 Glob/Grep으로 스캔하여 문서가 놓친 영향 지점을 탐색
- 문서가 전제로 삼는 코드 상태(함수 시그니처, 데이터 구조, 외부 의존성)를 Read로 확인
- 유사 패턴이 다른 곳에도 있는데 문서가 한쪽만 다루는 경우를 포착

능동적 확인이 불가능한 경우에만 WARN으로 표기한다.

### 개선 제안의 운영 원칙

각 모드는 체크리스트 평가와 별개로 "개선 제안" 섹션을 가진다. **옵션이다.** 제안할 게 없으면 "해당 없음"만 적는다. 채우려고 억지로 찾지 않는다.

평가의 1차 목적은 판정이다. 개선 제안은 보너스이며, 부실한 제안은 오히려 원 문서를 오염시킨다. 따라서 다음 네 조건을 **전부 만족할 때만** 제안한다:

1. 스펙/계획/구현의 **완전성·정확성·안전성**을 개선한다. 취향·스타일·선호 네이밍은 제외.
2. **구체적 근거로 왜 더 나은지** 설명할 수 있다 — 현재 vs 제안 비교, 발생 가능한 실패 시나리오, 관련 코드 증거 중 하나 이상.
3. **원 문서의 범위를 벗어나지 않는다** — 스코프 확장성 제안("이것도 함께 하면 좋겠다")은 제외.
4. **불확실하면 제안하지 않는다**. 사용자의 선호에 달려 있거나 trade-off가 불분명하면 의도적으로 비워 둔다.

출력 포맷:

```
### 개선 제안
- 대상: <문서/코드의 어느 부분>
- 현재: <현 내용 — 짧게 인용>
- 제안: <변경 내용>
- 근거: <왜 더 나은가 — 구체적 실패 시나리오 또는 코드 증거>
- 영향: <수용 시 예상 영향 범위>
```

해당 없으면:

```
### 개선 제안
해당 없음.
```

## mode: spec

스펙 문서의 품질과 완전성을 평가한다.

```markdown
### 1. 구현 착수 가능성
- [ ] 이 문서만 보고 추가 질문 없이 구현을 시작할 수 있는가
- [ ] 구현 방향이 하나로 특정되는가 (해석의 여지가 없는가)

### 2. AC 품질
- [ ] 모든 AC가 "~하면 ~한다" 형태로 검증 가능한가
- [ ] 각 AC에 검증 방법(테스트/수동확인 등)이 명시되었는가
- [ ] AC 간 모순이 없는가

### 3. 요구사항 완전성
- [ ] 에러/예외 시나리오가 다뤄졌는가
- [ ] 경계값/빈값 처리가 명시되었는가
- [ ] 동시성 이슈가 해당되는데 누락되지 않았는가

### 4. 코드베이스 정합성 (기존 코드 변경 시 — 직접 확인 필수)
- [ ] 스펙이 주장하는 현행 상태가 실제 코드와 일치하는가 (Grep/Read로 확인)
- [ ] 스펙이 명시적 주장을 하지 않더라도, 대상 영역 주변 모듈을 스캔하여 누락된 영향 지점을 찾았는가
- [ ] 변경 대상 외에 동일 패턴을 사용하는 지점이 있는지 Grep으로 조사했는가
- [ ] 영향 범위가 호출 체인까지 분석되었는가 (직접 추적 필수)
- [ ] 스펙이 전제하는 코드 상태(함수 시그니처, 데이터 구조, 외부 의존성)가 현재 코드와 일치하는가
- [ ] 기존 테스트 영향이 검토되었는가

### 5. 운영 관점
- [ ] migration/backfill이 필요한데 빠뜨리지 않았는가
- [ ] observability(로깅/모니터링)가 고려되었는가
- [ ] rollback/fallback 전략이 필요한데 빠뜨리지 않았는가

### 6. 가정 검증
- [ ] 문서에 근거 없이 "당연히" 전제된 가정이 없는가
- [ ] ambiguity ledger가 있다면, blocking/assumed 항목이 스펙에 confirmed처럼 반영된 곳이 없는가
- [ ] [TBD]나 미결정 사항이 구현 착수를 블로킹하는가

### 7. 개선 제안 (해당 시에만)
공통 원칙의 "개선 제안의 운영 원칙"을 따른다. 스펙이 더 완전·정확·안전해지는 제안이 있을 때만 작성한다. 없으면 "해당 없음".
```

## mode: plan

구현 계획의 완전성과 스펙 정합성을 평가한다.

```markdown
### 1. AC 커버리지
- [ ] 모든 AC가 하나 이상의 작업 단계에 매핑되는가
- [ ] 매핑되지 않은 AC가 없는가
- [ ] traceability matrix가 있다면, 빈 칸이 없는가

### 2. 작업 순서
- [ ] 의존 관계가 올바르게 반영된 순서인가
- [ ] 병렬 가능한 작업이 식별되었는가
- [ ] 순환 의존성이 없는가

### 3. 검증 계획
- [ ] 각 작업 단계에 검증 방법이 있는가
- [ ] 테스트 전략이 스펙 AC 기준인가 (구현 기준이 아닌)

### 4. 영향 범위 및 포함 코드 (능동적으로 확인)
- [ ] 수정 대상 파일이 구체적으로 명시되었는가
- [ ] 명시된 파일이 실제로 존재하는가 (Glob으로 확인)
- [ ] 계획에서 누락된 관련 파일이 없는가 (Grep으로 호출 체인 확인)
- [ ] 사이드이펙트 분석이 추측이 아닌 실제 호출체인·공유상태·트랜잭션 경계 추적으로 확인되었는가
- [ ] 계획에 코드 스니펫/API 호출/함수 시그니처가 포함되어 있다면, 실제 코드베이스와 일치하는가 (존재하지 않는 API, 잘못된 시그니처, 타입 불일치 여부)
- [ ] 계획이 도입하려는 의존성/패턴이 기존 코드와 충돌하지 않는가

### 5. 스펙 정합성
- [ ] 계획이 스펙 범위를 벗어나는 작업을 포함하지 않는가
- [ ] 스펙의 제약조건이 계획에 반영되었는가
- [ ] 스펙의 구현 지침이 계획에 반영되었는가

### 6. 개선 제안 (해당 시에만)
공통 원칙의 "개선 제안의 운영 원칙"을 따른다. 계획이 더 안전·정확해지는 제안(간과된 사이드이펙트, 더 안전한 작업 순서, 누락된 영향 파일 등)이 있을 때만 작성한다. 없으면 "해당 없음".
```

## mode: implementation

구현 결과가 스펙을 만족하는지, 그리고 코드 자체가 올바른지 평가한다.

```markdown
### 1. 스펙 준수
- [ ] 모든 AC가 구현되었는가 (누락 없음)
- [ ] 스펙에 없는 기능이 추가되지 않았는가 (과구현 없음)
- [ ] 스펙의 제약조건이 지켜졌는가

### 2. 테스트
- [ ] 모든 AC에 대응하는 테스트가 있는가
- [ ] 테스트가 스펙 요구사항 기준으로 작성되었는가 (구현 기준 아닌)
- [ ] 기존 테스트가 깨지지 않았는가 (회귀 없음)

### 3. 코드 품질 (스펙 준수와 별개로, 코드 자체의 정확성을 판단한다)
스펙 준수 체크만으로 끝내지 않는다. 변경 코드를 직접 읽고 다음을 확인한다.
- [ ] 기존 패턴/컨벤션과 일치하는가 (Read로 확인)
- [ ] 변경 지점의 호출 체인을 Grep으로 추적하여 사이드이펙트를 확인했는가
- [ ] 에러 핸들링이 스펙에 명시된 대로이며 누락된 오류 경로가 없는가
- [ ] 로직 정확성: 엣지케이스(빈 값/경계값/null/undefined), 조건 분기의 off-by-one·누락 분기
- [ ] 동시성/상태: race condition, 공유 상태 접근, 순서 의존성
- [ ] 리소스 관리: 파일·커넥션·락의 해제, 메모리 누수 징후, 큰 루프의 성능
- [ ] 보안: 입력 신뢰 가정의 적절성, injection 경로, secret/민감 데이터 유출
- [ ] 명백한 버그 징후: 중복 로직, dead code, 불일치한 타입 단언, 잘못 참조된 식별자

### 4. 운영 준비
- [ ] migration이 필요하면 작성되었는가
- [ ] 로깅/에러핸들링이 기존 패턴대로인가
- [ ] rollback이 필요하면 준비되었는가

### 5. 개선 제안 (해당 시에만)
공통 원칙의 "개선 제안의 운영 원칙"을 따른다. 스펙 준수와 별개로 코드 자체의 안전성·정확성을 개선하는 제안이 있을 때만 작성한다. 없으면 "해당 없음".
```

## mode: custom

호출자가 자유 형식으로 평가 기준을 작성한다.
체크리스트 자동 삽입 없이, 요청 파일의 Evaluation Criteria 섹션에 직접 기재한다.
