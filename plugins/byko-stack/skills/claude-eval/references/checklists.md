# 모드별 평가 체크리스트

## 공통 원칙: 코드베이스 검증

모든 모드에서, 평가 대상 문서가 코드베이스에 대한 주장을 포함하면 Glob/Grep/Read로 실제 코드를 확인하여 사실 여부를 검증한다. 문서만 보고 판단하지 않는다.

- 문서가 "X가 N곳에서 사용된다"고 주장하면 → Grep으로 실제 사용처를 세어 확인
- 문서가 "기존 패턴은 X다"고 주장하면 → 실제 코드를 읽어 확인
- 문서가 "영향 범위는 A, B다"고 주장하면 → 호출 체인을 추적하여 누락이 없는지 확인

---

## mode: spec

스펙 문서의 품질과 완전성을 평가한다.

```markdown
### 1. 구현 착수 가능성
- [ ] 이 문서만 보고 추가 질문 없이 구현을 시작할 수 있는가
- [ ] 구현 방향이 하나로 특정되는가 (해석의 여지가 없는가)

### 2. AC 품질
- [ ] 모든 AC가 "~하면 ~한다" 형태로 검증 가능한가
- [ ] 각 AC에 검증 방법(테스트/수동확인 등)이 명시되었는가
- [ ] AC 간 모순이 없는가

### 3. 요구사항 완전성
- [ ] 에러/예외 시나리오가 다뤄졌는가
- [ ] 경계값/빈값 처리가 명시되었는가
- [ ] 동시성 이슈가 해당되는데 누락되지 않았는가

### 4. 코드베이스 정합성 (기존 코드 변경 시)
- [ ] 스펙이 주장하는 현행 상태가 실제 코드와 일치하는가 (Grep/Read로 확인)
- [ ] 변경 대상으로 명시된 곳 외에 동일 패턴을 사용하는 누락된 지점이 없는가
- [ ] 영향 범위가 호출 체인까지 분석되었는가
- [ ] 기존 테스트 영향이 검토되었는가

### 5. 운영 관점
- [ ] migration/backfill이 필요한데 빠뜨리지 않았는가
- [ ] observability(로깅/모니터링)가 고려되었는가
- [ ] rollback/fallback 전략이 필요한데 빠뜨리지 않았는가

### 6. 가정 검증
- [ ] 문서에 근거 없이 "당연히" 전제된 가정이 없는가
- [ ] ambiguity ledger가 있다면, blocking/assumed 항목이 스펙에 confirmed처럼 반영된 곳이 없는가
- [ ] [TBD]나 미결정 사항이 구현 착수를 블로킹하는가
```

## mode: plan

구현 계획의 완전성과 스펙 정합성을 평가한다.

```markdown
### 1. AC 커버리지
- [ ] 모든 AC가 하나 이상의 작업 단계에 매핑되는가
- [ ] 매핑되지 않은 AC가 없는가
- [ ] traceability matrix가 있다면, 빈 칸이 없는가

### 2. 작업 순서
- [ ] 의존 관계가 올바르게 반영된 순서인가
- [ ] 병렬 가능한 작업이 식별되었는가
- [ ] 순환 의존성이 없는가

### 3. 검증 계획
- [ ] 각 작업 단계에 검증 방법이 있는가
- [ ] 테스트 전략이 스펙 AC 기준인가 (구현 기준이 아닌)

### 4. 영향 범위 (코드 검증)
- [ ] 수정 대상 파일이 구체적으로 명시되었는가
- [ ] 명시된 파일이 실제로 존재하는가 (Glob으로 확인)
- [ ] 계획에서 누락된 관련 파일이 없는가 (Grep으로 호출 체인 확인)
- [ ] 사이드이펙트 분석이 포함되었는가

### 5. 스펙 정합성
- [ ] 계획이 스펙 범위를 벗어나는 작업을 포함하지 않는가
- [ ] 스펙의 제약조건이 계획에 반영되었는가
- [ ] 스펙의 구현 지침이 계획에 반영되었는가
```

## mode: implementation

구현 결과가 스펙을 정확히 만족하는지 평가한다.

```markdown
### 1. 스펙 준수
- [ ] 모든 AC가 구현되었는가 (누락 없음)
- [ ] 스펙에 없는 기능이 추가되지 않았는가 (과구현 없음)
- [ ] 스펙의 제약조건이 지켜졌는가

### 2. 테스트
- [ ] 모든 AC에 대응하는 테스트가 있는가
- [ ] 테스트가 스펙 요구사항 기준으로 작성되었는가 (구현 기준 아닌)
- [ ] 기존 테스트가 깨지지 않았는가 (회귀 없음)

### 3. 코드 품질 (코드 검증)
- [ ] 변경된 코드를 Read로 읽어 기존 패턴/컨벤션과 일치하는지 확인
- [ ] 변경 지점의 호출 체인을 Grep으로 추적하여 사이드이펙트 확인
- [ ] 에러 핸들링이 스펙에 명시된 대로인가

### 4. 운영 준비
- [ ] migration이 필요하면 작성되었는가
- [ ] 로깅/에러핸들링이 기존 패턴대로인가
- [ ] rollback이 필요하면 준비되었는가
```

## mode: custom

호출자가 자유 형식으로 평가 기준을 작성한다.
체크리스트 자동 삽입 없이, 요청 파일의 Evaluation Criteria 섹션에 직접 기재한다.
