# Spec Designer

유저와의 **대화**를 통해 구현 스펙을 완성한다. 스펙은 이후 에이전트의 유일한 지침서 역할을 한다.

핵심 원칙: **모호한 상태를 통과시키지 않는다.** 모르면 물어보고, 코드에서 확인할 수 있으면 확인 후 유저에게 검증받는다.

---

## 이 스킬이 트리거되면

### Step 1: 첫 응답 — 코드를 먼저 살피고, 맥락 위에서 질문한다

유저의 요청을 읽은 뒤, 질문하기 전에 코드베이스에서 관련 정보를 먼저 수집한다. 코드에서 답을 찾을 수 있는 것은 직접 확인하고, 유저에게는 코드로 판단할 수 없는 것만 묻는다.

**코드베이스가 있는 경우 (브라운필드):**

먼저 요청이 코드 탐색에 충분히 구체적인지 판단한다. 변경 대상이나 도메인이 특정되어야 의미 있는 탐색이 가능하다.

**요청이 구체적일 때** (변경 대상, 도메인, 기능이 특정 가능):
1. 요청과 관련된 코드를 빠르게 탐색한다 (Glob/Grep/Read)
   - 요청이 언급하는 도메인, 기능, 모듈이 이미 존재하는가
   - 유사한 기능이 어떻게 구현되어 있는가 (패턴, 컨벤션)
   - 프로젝트 구조와 기술 스택
2. 탐색 결과를 기반으로 첫 응답을 구성한다:
   - **코드에서 확인한 사실** — 기존 구현, 패턴, 관련 코드 위치 (파일:라인 근거)
   - **코드 사실 위의 확인 질문** — "X 패턴을 쓰고 있는데, 이번에도 같은 방식으로 갈까요?"
   - **코드로 답할 수 없는 질문** — 제품 방향, 비즈니스 정책 등 유저만 결정할 수 있는 사항

**요청이 넓거나 모호할 때** (어디를 탐색해야 할지 특정할 수 없음):
1. 요청에서 파악한 것을 정리한다
2. 탐색 범위를 좁히기 위한 질문을 먼저 한다
   - "이 기능이 기존 X 모듈의 확장인가요, 새로운 영역인가요?"처럼 코드 탐색 대상을 특정할 수 있는 질문
3. 유저 답변으로 범위가 좁혀지면 Step 2에서 코드 탐색을 시작한다

질문 라우팅(`references/question-routing.md`)은 이 단계부터 적용한다. Type A(코드 확인 가능)는 직접 확인하고 confirmation만 구하고, Type B(유저 결정)와 Type D(정보 부재)만 열린 질문으로 던진다.

**코드베이스가 없는 경우 (그린필드):**

1. 요청에서 파악한 것을 간단히 정리한다 (3-5줄)
2. 핵심 결정 사항 2-3개를 질문한다

이 단계에서 파일을 생성하지 않는다.

**질문 전략:**
- 한번에 3개 이상 질문하지 않는다 — 대화가 부담스러워지면 유저가 대충 답한다
- "혹시 ~도 필요한가요?"보다 "~는 어떻게 처리할 생각이세요?"가 더 구체적인 답을 끌어낸다
- 유저가 "나중에 정하자"는 부분은 기억해두고 나중에 `[TBD]`로 남긴다

**외부 참고자료 처리:**
- Linear 이슈가 언급되면 Linear MCP로 가져온다
- 외부 API 문서 URL이 있으면 WebFetch로 참고한다 — 단, 외부는 참고일 뿐 우리 시스템 표준을 정립하는 게 핵심
- 관련 내부 문서(`docs/` 하위)가 있으면 먼저 읽고 컨텍스트를 확보한다

⏸️ **유저 답변을 기다린다.**

---

### Step 2: 심층 코드 분석 + 추가 질문

유저 답변을 반영한 뒤, Step 1의 초기 탐색을 넘어 연쇄 영향과 숨겨진 의존성까지 심층 분석한다.

**심층 분석의 원칙: 유저가 요청한 변경 지점만 보지 않는다.**

요청된 코드의 연쇄 영향을 반드시 추적한다. 이 추적에서 유저가 인식하지 못한 관련 수정이 발견되면, 코드 근거와 함께 유저에게 제시한다.

**분석 순서:**

1. **요청 지점** — 유저가 언급한 코드를 읽는다
2. **호출 체인** — 해당 코드를 호출하는 곳(callers)과 호출당하는 곳(callees)을 따라간다
3. **개념 수준 탐색** — 특정 함수명 검색에 그치지 않는다. 관련 필드명·에러 코드·도메인 용어로 검색을 넓혀, 같은 개념이 다른 메서드명으로 구현된 곳까지 찾는다
4. **연산 완전성** — 특정 도메인 엔티티의 일부 연산에서 패턴을 발견하면, 해당 엔티티의 나머지 연산도 빠짐없이 확인한다
5. **영향 범위** — 변경 시 깨질 수 있는 기존 코드, 테스트, 공유 상태를 확인한다

**추가로 확인하는 것:**
- 프로젝트 구조 및 기존 코드 패턴 (컨벤션, 에러 핸들링)
- 테스트 현황 (프레임워크, 패턴, 커버리지)
- 심화: 데이터 흐름, DB 스키마, 외부 서비스 연동 (목표가 요구할 때)

분석 범위가 넓으면 subagent로 병렬 분석한다.

**분석 결과를 유저에게 보여줄 때:**

유저가 요청한 것과 추적에서 추가로 발견한 것을 구분하여 제시한다:

- **요청 범위**: 유저가 언급한 코드에서 확인한 사실
- **추가 발견**: 호출 체인/동일 패턴 추적에서 발견한 연쇄 영향 — 반드시 파일:라인 근거와 함께

코드 근거 없이 범위를 확장하지 않는다.
발견이 없으면 억지로 만들어내지 않는다 — 유저의 원래 범위가 적절할 수도 있다.

**질문 라우팅 규칙** (상세: `references/question-routing.md`):

| Type | 판별 | 행동 |
|------|------|------|
| **A** | 코드에서 확인 가능 | 코드를 읽고, 유저에게 confirmation만 |
| **B** | 유저만 결정 가능 (제품/정책/우선순위) | 반드시 유저에게 질문. **절대 AI가 자체 결정 금지** |
| **C** | 코드 사실 + 판단 혼합 | 사실 먼저 정리 → 판단만 질문 |
| **D** | 판단 필요한데 정보 없음 | **절대 자체 생성 금지**. 유저에게 반드시 질문 |

⏸️ **유저 답변을 기다린다.**

### Step 3: 반복 — 충분한 정보가 모일 때까지

Step 2를 반복한다. 매 라운드마다:

1. 유저 답변을 반영한다
2. 필요하면 코드를 추가 분석한다
3. 아직 모호한 것이 있으면 질문한다

**이 반복을 멈추는 시점:**

다음 13개 항목 대부분에 대해 답을 알고 있으면 (유저 확인 or 코드 확인):

> goal, scope/non-goals, current-state facts, desired behavior, data model,
> external interfaces, failure handling, auth/permission, migration/backfill,
> observability/logging, rollout/fallback, acceptance criteria, verification method

"대부분"의 기준: **blocking 0개, 모르는 것 3개 이하**. 상세 항목 설명은 `references/ambiguity-ledger.md` 참조.

⏸️ **매 라운드마다 유저 답변을 기다린다.**

---

### Step 4: Ambiguity Ledger 생성 + 스펙 초안

정보가 충분히 모이면 두 파일을 생성한다:

1. **ambiguity-ledger.md** — 13개 항목의 현재 상태 (상세 템플릿: `references/ambiguity-ledger.md`)
2. **spec.md** 초안 (상세 템플릿: `references/spec-templates.md`)

**파일 위치:** `docs/specs/[project-name]/`
- `[project-name]`은 프로젝트 성격에 맞는 간결한 영문 디렉토리명

**범위 판단:**
- 독립 기능 단위 3개 이상, 서로 다른 도메인 → 멀티 스펙 (메인 + 서브)
- 그 외 → 단일 스펙
- 분할 필요 시 유저에게 제안하고 합의 후 진행

**ledger 게이트 확인:**

| 상태 | 의미 | 게이트 |
|------|------|--------|
| `confirmed` | 유저가 확인함 | 통과 |
| `from-code` | 코드에서 확인 (파일:라인 근거) | 통과 |
| `assumed` | AI 추론, 유저 미확인 | **2개 이하만** |
| `open` | 아직 미논의 | **3개 이하만** |
| `blocking` | 해소 전 진행 불가 | **0개여야 함** |

게이트 미통과 시 → blocking/open 항목에 대해 유저에게 추가 질문 후 Step 3으로 돌아간다.

**스펙 초안 필수 섹션:**
- 목표, 배경, 참고자료, 현행 분석 요약
- 도메인 규칙, 구현 상세
- **Acceptance Criteria (AC) 테이블** — 단일 진실 공급원
- 테스트 전략, 이슈 및 결정 사항, 구현 순서, 구현 지침

**AC 작성 규칙:**
- "~가 동작한다" (X) → "~를 입력하면 ~가 반환된다" (O) — 검증 가능한 형태로
- 각 AC에 검증 방법 필수
- 검증 방법 없는 AC → 유저에게 질문

유저에게 ledger 상태와 스펙 초안을 보여준다.

⏸️ **유저 피드백을 기다린다.**

---

### Step 5: 반복 — 유저가 만족할 때까지

유저 피드백을 받고:
1. 스펙을 수정한다
2. ledger를 업데이트한다
3. 유저에게 다시 보여준다

**유저가 만족하면**, 다음 단계를 묻는다:

📍 **"스펙 초안이 완성되었습니다. 다음 중 어떻게 진행할까요?"**
- `/eval-gate spec` — 독립 컨텍스트에서 스펙 품질 검증 (같은 대화 내 자기 평가는 편향이 있을 수 있음)
- `/spec-dev` — 검증 없이 바로 구현 시작

⏸️ **유저 답변을 기다린다.**

## 중단 후 재개

`docs/specs/[project-name]/` 아래에 spec.md와 ambiguity-ledger.md가 이미 있으면:

1. ledger에서 현재 게이트 상태를 확인한다
2. 스펙의 진행 현황을 파악한다
3. 유저에게 현재 상황을 요약하고 다음 단계를 제안한다

---

## 에이전트 행동 지침

1. **깊이있게 사고한다** — 표면적 분석이 아니라, 왜 그런 패턴인지, 어떤 트레이드오프가 있는지까지 고려한다. 코드를 읽을 때 "무엇"뿐 아니라 "왜"도 파악한다. 엣지케이스, 동시성 이슈, 실패 시나리오까지 고려하여 스펙에 반영한다.
2. **첫 응답부터 코드를 살핀다** — 코드베이스가 있으면 질문 전에 관련 코드를 탐색하여, 코드로 확인 가능한 것은 직접 확인하고 유저에게는 판단이 필요한 것만 질문한다. 파일 생성이나 스펙 작성은 첫 응답에서 하지 않는다
3. **매 응답이 유저에게 무언가를 묻거나 보여주고 확인을 구한다** — 유저 상호작용 없이 다음 단계로 넘어가지 않는다
4. **요청 지점 너머를 탐색한다** — 호출 체인, 동일 패턴, 영향 범위를 추적하여 코드 근거로 범위 확장이 필요하면 제시한다. 이것이 코드 분석의 가장 중요한 역할이다.
5. **질문 라우팅을 반드시 따른다** — 모호한데 자체 결정하지 않는다 (상세: `references/question-routing.md`)
6. **게이트를 우회하지 않는다** — blocking이 있으면 스펙 작성 금지. 유저가 "그냥 진행해"라고 해도 영향을 설명하고 `open`으로 전환할지 확인받는다
7. **코드 분석은 subagent를 활용한다** — 분석 범위가 넓을 때 도메인별 병렬 분석
8. **기존 패턴을 존중한다** — 코드베이스의 컨벤션을 분석하고, 스펙이 이를 따르도록 한다
