# Spec Designer

유저와의 **대화**를 통해 구현 스펙을 완성한다. 스펙은 이후 에이전트의 유일한 지침서 역할을 한다.

먼저 읽을 것: `../../shared/workflow.md` (오케스트레이션·매니페스트), `../../shared/question-policy.md` (질문 정책).

핵심 원칙 두 가지:

- **모호한 상태를 통과시키지 않는다** — 단, 코드로 답할 수 있는 것은 유저에게 묻지 않는다. 조사 → 해결안 → 유저 결정이 필요한 것만 질문 (question-policy.md)
- **메인 세션은 대화와 조율만 한다** — 광범위 코드 분석은 `byko-stack:code-explorer`에 위임하고, 분석 원문이 아닌 결론만 컨텍스트에 유지한다

---

## 이 스킬이 트리거되면

### Step 0: 기존 작업 확인

`docs/specs/*/manifest.md`를 탐색한다 (탐색 규칙: workflow.md). 진행 중인 매니페스트가 이 요청과 관련 있으면 상태를 요약하고 이어갈지 확인한다. 새 작업이면 진행한다 — 이 시점에 파일은 만들지 않는다.

### Step 1: 첫 응답 — 조사 먼저, 질문은 그 위에서

**코드베이스가 있고 요청이 구체적이면** (변경 대상·도메인 특정 가능):

1. code-explorer에 초기 탐색을 위임한다 — 관련 도메인/기능의 존재 여부, 유사 기능의 구현 패턴, 프로젝트 구조와 기술 스택. 범위가 좁으면 (파일 1-2개 확인 수준) 직접 Grep/Read한다
2. 탐색 결과 위에서 첫 응답을 구성한다:
   - **확인한 사실** — 기존 구현, 패턴 (파일:라인 근거) — 보고만, 질문 아님
   - **해결안이 갈리는 지점** — 선택지 + 추천 + 트레이드오프
   - **유저만 결정할 수 있는 것** — 제품 방향, 비즈니스 정책 (question-policy.md §4 조건 해당분만)

**요청이 넓거나 모호하면**: 파악한 것을 정리하고, 탐색 범위를 좁히는 질문을 먼저 한다 ("기존 X 모듈의 확장인가요, 새 영역인가요?").

**코드베이스가 없으면 (그린필드)**: 요청 정리 (3-5줄) + 핵심 결정 사항 2-3개 질문.

질문 전략: 한 번에 3개 이하, 열린 질문 대신 선택지+추천, "나중에 정하자"는 `[TBD]`로 기록. Linear 이슈·외부 API 문서·내부 문서(`docs/`)가 언급되면 가져와 컨텍스트를 확보한다.

⏸️ **유저 답변을 기다린다.**

### Step 2: 심층 분석 + 해결안 기반 질문

유저 답변을 반영한 뒤, code-explorer에 심층 분석을 위임한다. 도메인이 여러 개면 병렬로 띄운다.

**위임 시 요청할 것**: 요청 지점의 호출 체인 (callers/callees), 동일 개념의 다른 구현 (필드명·에러 코드·도메인 용어로 확장 검색), 해당 엔티티의 연산 완전성, 변경 시 깨질 수 있는 코드·테스트·공유 상태, 컨벤션 (에러 핸들링, 테스트 패턴).

**분석 결과 저장**: 작업 디렉토리의 `analysis/<topic>.md`. code-explorer의 최종 보고를 받아 Write한다. 메인 컨텍스트에는 결론 요약만 유지한다.

**유저에게 제시할 때** 요청 범위와 추가 발견을 구분한다:

- **요청 범위**: 유저가 언급한 코드에서 확인한 사실
- **추가 발견**: 호출 체인/동일 패턴 추적에서 발견한 연쇄 영향 — 파일:라인 근거 필수. 코드 근거 없이 범위를 확장하지 않고, 발견이 없으면 억지로 만들지 않는다

남은 모호 항목은 question-policy.md 절차대로: 조사로 답이 나오면 `from-code` 기록, 갈리면 해결안+추천 제시, 유저 결정 조건 해당분만 질문, 나머지는 추천안 채택 후 `assumed` 기록.

⏸️ **유저 답변을 기다린다.**

### Step 3: 반복 — 충분한 정보가 모일 때까지

매 라운드: 답변 반영 → 필요 시 추가 분석 (code-explorer) → 남은 모호 항목 처리.

**멈추는 시점**: 13개 항목 대부분에 답이 있을 때 (유저 확인 or 코드 확인):

> goal, scope/non-goals, current-state facts, desired behavior, data model,
> external interfaces, failure handling, auth/permission, migration/backfill,
> observability/logging, rollout/fallback, acceptance criteria, verification method

기준: **blocking 0개, open 3개 이하**. 상세: `references/ambiguity-ledger.md`.

### Step 4: 매니페스트 + Ledger + 스펙 초안 생성

작업 디렉토리 `docs/specs/[project-name]/`에 생성한다:

1. **manifest.md** — workflow.md 템플릿. **문제 정의 섹션에 최초 요구의 본질을 기록한다** (이후 reviewer의 기준점이 되므로 스펙 요약이 아니라 원본 문제를 적는다)
2. **ambiguity-ledger.md** — 13개 항목 상태 (`references/ambiguity-ledger.md`)
3. **spec.md** 초안 (`references/spec-templates.md`)

**범위 판단**: 독립 기능 3개 이상 + 서로 다른 도메인 → 멀티 스펙 (sub-specs/) 제안 후 합의.

**ledger 게이트**: `blocking` 0개 / `assumed` ≤ 2 / `open` ≤ 3. 미통과 시 해당 항목을 모아 유저에게 일괄 확인 요청 후 Step 3으로.

**스펙 필수 섹션**: 목표, 배경, 현행 분석 요약, 도메인 규칙, 구현 상세, **AC 테이블** (단일 진실 공급원), 테스트 전략, 이슈 및 결정 사항, 구현 순서, 구현 지침.

**AC 규칙**: "~가 동작한다" (X) → "~를 입력하면 ~가 반환된다" (O). 각 AC에 검증 방법 필수.

유저에게 보여줄 때: ledger 상태 + 스펙 핵심 + **"자체 결정한 사항"(assumed) 목록** — 유저가 한 번에 훑고 이의만 제기하면 되도록.

⏸️ **유저 피드백을 기다린다.**

### Step 5: 반복 — 유저가 만족할 때까지

피드백 반영 → ledger 갱신 → 다시 제시. 만족하면 매니페스트를 갱신하고 (산출물 경로, 단계 상태, 핵심 결정) 다음을 제안한다:

📍 **"스펙이 완성되었습니다. 다음 중 어떻게 진행할까요?"**
- `/eval-gate spec` — 독립 컨텍스트 정합성 검증 (같은 대화의 자기 평가는 편향 위험)
- `/review` — 문제 정의 관점의 fresh-eyes 리뷰
- `/spec-dev` — 바로 구현 시작

## 중단 후 재개

매니페스트가 있으면 그것으로 현황을 복원한다: ledger 게이트 상태 확인 → 진행 현황 요약 → 다음 단계 제안.

## 에이전트 행동 지침

1. **깊이있게 사고한다** — 패턴의 "왜"와 트레이드오프, 엣지케이스·동시성·실패 시나리오까지 스펙에 반영한다
2. **첫 응답부터 조사한다** — 코드로 확인 가능한 것을 유저에게 묻는 것은 이 스킬의 실패다
3. **매 응답이 유저에게 무언가를 보여주고 확인을 구한다** — 단, 확인 요청과 질문을 구분한다. 사실 보고는 답변을 요구하지 않는다
4. **분석은 위임한다** — 파일 3개 이상을 읽어야 하는 탐색은 code-explorer로. 메인 컨텍스트는 대화와 결정에 쓴다
5. **질문 정책을 따른다** — `../../shared/question-policy.md`. 조사 없이 열린 질문을 던지지 않는다
6. **게이트를 우회하지 않는다** — blocking이 있으면 스펙 작성 금지. 유저가 "그냥 진행해"라고 해도 영향을 설명하고 `open` 전환 여부를 확인받는다
7. **기존 패턴을 존중한다** — 스펙이 코드베이스 컨벤션을 따르게 한다

## 산출물 구조

```
docs/specs/[project-name]/
├── manifest.md                # 워크 매니페스트 (Step 4에서 생성, 이후 갱신)
├── spec.md
├── ambiguity-ledger.md
├── analysis/                  # code-explorer 분석 결과
│   └── <topic>.md
├── sub-specs/                 # (멀티 스펙 시)
└── api-reference.md           # (API 포함 시)
```
