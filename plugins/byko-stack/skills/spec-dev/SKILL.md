# Spec-Dev: 스펙 기반 구현

스펙 문서를 기반으로 실제 코드를 구현한다.
핵심 원칙: **스펙에 쓰인 대로, 기존 코드 스타일대로, 사이드이펙트 없이** 구현한다.
스펙을 무조건 신뢰하지 않는다 — 구현 전에 스펙의 품질을 검사하고, 부족하면 되돌린다.

---

## 이 스킬이 트리거되면

### Step 1: 스펙 읽기 + 품질 확인

유저가 스펙 경로를 제공하면 읽는다. 경로가 없으면 `docs/specs/` 아래에서 찾는다.

**읽을 파일:** spec.md, ambiguity-ledger.md, codebase-analysis.md (있으면), sub-specs/ (있으면)

**품질 확인 (하나라도 실패 시 구현을 거부한다):**

1. ambiguity-ledger.md에 `blocking` 항목이 있는가?
   - 있으면 → "ledger에 blocking 항목이 있습니다. `/spec-designer`로 해소해주세요."
2. eval-gate 결과가 있는가? (`eval-results/spec-gate-*.result.md` glob)
   - 있는데 NEEDS_REVISION → "스펙 검증에서 수정 필요 판정을 받았습니다. 수정 후 재검증해주세요."
   - 없으면 → 경고만 하고 유저에게 진행 여부 확인 (eval-gate 없이도 진행 가능)
3. AC 테이블에 모든 AC의 검증 방법이 있는가?
   - 없으면 → "검증 방법이 없는 AC가 있습니다. 스펙을 보완해주세요."

품질 확인 통과 후, 유저에게 간단히 확인한다:
- 구현 목표 요약
- 주요 AC 항목
- `[TBD]` 항목이 있으면 이 시점에서 결정을 요청한다

⏸️ **유저 확인을 기다린다.**

---

### Step 2: 구현 분석

스펙을 이해한 상태에서, **구현 관점**의 코드베이스 분석을 수행한다.
codebase-analysis.md가 이미 있으면 활용하되, 구현에 필요한 추가 분석만 보충한다.

**분석 항목:**
- 구현 대상의 기존 컨벤션 (네이밍, 구조, 패턴)
- 변경이 필요한 기존 코드와 영향 범위
- 테스트 패턴 (프레임워크, 디렉토리, mock/fixture 방식)
- 필요시: 의존 관계, DB 스키마, 외부 서비스 구조

분석 범위가 넓으면 에이전트로 병렬 분석한다.

분석 결과를 유저에게 보여주고, 구현 전략을 제안한다.

---

### Step 3: 전략 제안

스펙과 분석을 종합하여 유저에게 전략을 제안한다:

| 전략 | 적합한 경우 | 산출물 |
|------|------------|--------|
| **A: 직접 구현** | 변경 범위가 작고 명확 | 코드 + 테스트 |
| **B: 계획 후 구현** | 여러 단계 필요, 사이드이펙트 주의 | traceability.md → implementation-plan.md → 코드 + 테스트 |
| **C: Ralph-loop 준비** | 대규모 작업, 자율 반복이 효율적 | plan + traceability + progress.md + ralph-loop 프롬프트 |

추천 전략과 근거를 설명한다.

⏸️ **유저가 전략을 선택할 때까지 기다린다.**

---

### Step 4: 실행

#### 전략 A: 직접 구현

1. 스펙의 구현 순서에 따라 단계별로 구현한다
2. 각 단계 완료 후 관련 테스트를 실행하여 검증한다
3. 병렬 처리 가능한 단위는 에이전트에게 분배한다
4. 완료 후 Step 5(검증)로 이동한다

#### 전략 B: 계획 후 구현

1. **traceability.md 작성** — AC → 구현 Step → 테스트 매핑

   | AC ID | AC 내용 | 구현 Step | 대상 파일 | 테스트 | 상태 |
   |-------|---------|-----------|----------|--------|------|
   | AC-1 | ... | Step 1 | src/... | test/... | ⬜ |

   모든 AC가 매핑되어야 한다. 빈 칸이 있으면 plan 수정.

2. **implementation-plan.md 작성** — 실행 가능한 단계로 분해

3. 유저에게 plan을 보여준다.
   ⏸️ **유저 확인을 기다린다.**

4. 유저 승인 후:
   📍 **`/eval-gate plan docs/specs/[project-name]/implementation-plan.md`로 계획을 독립 검증할 수 있습니다.**
   유저가 건너뛰면 바로 구현으로 진행한다.

5. 구현 실행.
   - 단계별로 테스트 실행하여 검증
   - 완료 후 Step 5(검증)로 이동

#### 전략 C: Ralph-loop 준비

상세 산출물 템플릿은 `references/ralph-loop-template.md`를 읽고 따른다.

1. traceability.md + implementation-plan.md 작성 (전략 B와 동일)
2. progress.md 생성 (AC 기반 추적)
3. ralph-loop-prompt.md 생성 (자기완결적 프롬프트)
4. 유저에게 생성된 파일과 실행 커맨드 안내:

```
📍 실행 커맨드:
/ralph-loop "$(cat docs/specs/[project-name]/ralph-loop-prompt.md)" --max-iterations N --completion-promise "ALL_TASKS_COMPLETE"
```

⏸️ **유저에게 파일 확인 및 실행 안내 후 기다린다.**

---

### Step 5: 검증 (전략 A/B 완료 후)

1. **스펙 대조 검증**: AC 테이블 항목을 하나씩 대조하여 누락 확인
2. **테스트 실행**: 전체 관련 테스트 통과 확인
3. **사이드이펙트 확인**: 기존 테스트가 깨지지 않았는지 확인
4. **독립 검증 안내**: 유저에게 eval-gate 진행 여부를 묻는다

자체 AC 대조와 테스트 통과는 같은 컨텍스트 내의 확인이므로 편향이 있을 수 있다. 독립 컨텍스트 평가(eval-gate)를 거쳐야 놓친 문제가 드러난다.

검증 결과를 보여준 뒤, 유저에게 묻는다:

📍 **"구현 자체 검증은 통과했습니다. `/eval-gate implementation`으로 독립 검증을 진행할까요?"**

⏸️ **유저 답변을 기다린다.**

## Bounce-Back 규칙

구현 중 문제를 만나면 분류한다:

**코드 문제** (여기서 해결):
- 로직 버그, 테스트 실패, 통합 이슈 → 코드를 수정한다

**스펙 문제** (spec-designer로 되돌린다):
- AC가 모호하여 구현 방향을 결정할 수 없음
- 스펙에 누락된 결정 사항 발견
- AC 간 모순 발견
- 구조적으로 구현 불가능한 요구사항

📍 **스펙 문제 발견 시: "스펙에서 [문제]를 발견했습니다. `/spec-designer`로 돌아가서 보완이 필요합니다."**

## 구현 공통 지침

- **깊이있게 사고한다**: 표면적 구현이 아니라, 엣지케이스와 동시성 이슈까지 고려한다. 스펙의 "이슈 및 결정 사항" 섹션을 반드시 확인하고 반영한다. 코드를 읽을 때 "무엇"뿐 아니라 "왜"도 파악한다.
- **코드베이스 우선**: 기존 컨벤션(네이밍, 구조, 에러 핸들링)을 반드시 따른다
- **사이드이펙트 분석**: 변경이 기존 기능에 영향을 주는지 호출 체인을 추적하여 확인한다
- **테스트는 스펙 기준**: 코드가 아닌 AC 기준으로 테스트를 작성한다 (코드 맞춤 테스트는 버그를 통과시킨다)
- **스펙 준수**: 스펙에 없는 기능을 추가하지 않는다. `[TBD]` 항목은 유저에게 결정 요청
- **에이전트 활용**: 독립적 구현 단위가 있으면 병렬 처리
