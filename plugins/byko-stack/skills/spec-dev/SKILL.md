# Spec-Dev: 스펙 기반 구현

스펙을 기반으로 실제 코드를 구현한다. **스펙에 쓰인 대로, 기존 코드 스타일대로, 사이드이펙트 없이.**

먼저 읽을 것: `../../shared/workflow.md`, `../../shared/question-policy.md`.

메인 세션의 역할은 **구현 루프의 오케스트레이션**이다: 대상 파악 → 분석 위임 → 전략 결정 → 구현 단위 분배 → 검증 → 매니페스트 갱신. 큰 구현에서 코드 작성 자체는 `byko-stack:implementer`에 위임하여 메인 컨텍스트를 보존한다.

---

## 이 스킬이 트리거되면

### Step 1: 대상 파악 + 품질 확인

**대상 탐색** (workflow.md 규칙): 인자 > 매니페스트 탐색 > 유저에게 확인.

**Case A — 스펙이 있는 경우**: 매니페스트가 가리키는 spec.md, ambiguity-ledger.md, analysis/를 읽는다.

품질 확인 (실패 시 구현을 거부한다):

1. ledger에 `blocking` 항목 → "`/spec-designer`로 해소해주세요"
2. 매니페스트에 eval 결과가 NEEDS_REVISION → "수정 후 재검증해주세요". eval 기록이 없으면 경고만 하고 진행 여부 확인
3. 검증 방법 없는 AC → "스펙을 보완해주세요"

통과 후: 구현 목표와 주요 AC를 요약하고, `[TBD]` 항목이 있으면 이 시점에 결정을 요청한다 (선택지+추천 형태로).

**Case B — 스펙 없이 호출된 경우**: 막지 않는다. 대화에서 합의된 요구사항을 정리하여:

1. 목표·범위·**최소 AC 목록**(항목당 검증 방법 포함)을 유저에게 확인받는다
2. `docs/specs/<project>/manifest.md`를 경량으로 생성한다 — 문제 정의 + AC를 매니페스트에 직접 기록 (별도 spec.md 없이)
3. 진행한다. 단, 작업 중 범위가 계속 자라면 "/spec-designer로 정식 스펙을 만드는 게 안전합니다"를 제안한다

AC 없이는 진행하지 않는다 — AC가 곧 완료 조건이자 테스트 기준이다.

⏸️ **유저 확인을 기다린다.**

### Step 2: 구현 분석

구현 관점의 코드베이스 분석을 `byko-stack:code-explorer`에 위임한다. analysis/가 이미 있으면 활용하고 부족분만 보충한다.

**위임 시 요청할 것**: 구현 대상의 기존 컨벤션 (네이밍, 구조, 에러 핸들링), 변경 필요한 기존 코드와 영향 범위, 테스트 패턴 (프레임워크, 디렉토리, mock/fixture), 필요시 의존 관계·DB 스키마·외부 연동.

결과는 `analysis/impl-<topic>.md`로 저장하고, 메인 컨텍스트에는 결론만 유지한다. 도메인이 여러 개면 병렬 위임.

분석 결과를 유저에게 요약 제시한다.

### Step 3: 전략 결정

| 전략 | 적합한 경우 | 방식 |
|------|------------|------|
| **A: 직접 구현** | 변경 범위가 작고 명확 (파일 몇 개, AC 1-3개) | 메인 세션이 직접 구현 |
| **B: 계획 후 구현** | 여러 단계, 사이드이펙트 주의 필요 | traceability + plan 작성 → 직접 구현 |
| **C: 구현 루프** | AC가 많거나 독립 단위로 분해 가능한 큰 작업 | plan 작성 → implementer 위임 루프 |

추천 전략과 근거를 제시한다. 명확한 경우 (예: AC 2개짜리 소규모 수정 = A) 추천안으로 진행을 제안하고, 갈리는 경우만 선택을 기다린다.

### Step 4: 실행

#### 전략 A: 직접 구현

스펙의 구현 순서대로 단계별 구현 → 각 단계 후 관련 테스트 실행 → Step 5로.

#### 전략 B: 계획 후 구현

1. **traceability.md** — AC → 구현 Step → 대상 파일 → 테스트 매핑. 모든 AC가 매핑되어야 한다
2. **implementation-plan.md** — 실행 가능한 단계로 분해
3. 유저에게 plan 제시. ⏸️ 확인 대기. (`/eval-gate plan`으로 독립 검증 가능함을 안내)
4. 승인 후 구현 → Step 5로

#### 전략 C: 구현 루프 (오케스트레이션)

상세 프로토콜: `references/dev-loop.md`. 요약:

1. 전략 B와 동일하게 traceability + plan 작성, `progress.md` 생성
2. 유저 승인 후 루프 시작:
   - **선택**: progress에서 미완료 단위 중 의존성이 풀린 것들을 고른다
   - **분배**: 독립적인 단위들은 implementer를 병렬로 띄운다. 각 implementer에는 해당 AC 전문, 스펙 경로, 컨벤션 노트(analysis/), progress 경로를 전달한다
   - **수합**: 보고를 받아 검증한다 — 변경 파일 확인, 테스트 결과 확인. DONE이면 progress·traceability 갱신, BOUNCE_BACK이면 Bounce-Back 규칙으로
   - **방향 점검** (3라운드마다): 전체 테스트 실행, AC 커버리지 확인, plan과 실제의 괴리 확인. 의미 단위로 git commit
   - 모든 AC 완료 시 루프 종료 → Step 5로
3. 루프 중 메인 컨텍스트가 한계에 가까워지면: progress·매니페스트를 최신화하고 유저에게 알린다 — 매니페스트만으로 새 세션에서 `/spec-dev` 재개가 가능하다

### Step 5: 검증

1. **AC 대조**: AC 테이블을 하나씩 대조하여 누락 확인
2. **전체 테스트**: 관련 테스트 전체 통과 확인
3. **사이드이펙트**: 기존 테스트가 깨지지 않았는지 확인
4. **매니페스트 갱신**: 단계 상태, 변경 파일, 핵심 결정

자체 검증은 같은 컨텍스트의 확인이므로 편향이 있다. 결과를 보여준 뒤 제안한다:

📍 **"자체 검증 통과. 독립 검증을 진행할까요?"**
- `/eval-gate implementation` — 스펙 정합성 + 코드 정확성 평가
- `/review` — 문제 정의 관점 fresh-eyes 리뷰 (정합성을 넘어 방향·성실성 점검)

⏸️ **유저 답변을 기다린다.**

## Bounce-Back 규칙

구현 중 문제를 만나면 분류한다:

- **코드 문제** (로직 버그, 테스트 실패, 통합 이슈) → 여기서 수정한다
- **스펙 문제** (AC 모호로 방향 결정 불가, 누락된 결정, AC 간 모순, 구조적 구현 불가) → 코드로 우회하지 않는다. 매니페스트 미해결 섹션에 기록하고:

📍 **"스펙에서 [문제]를 발견했습니다. `/spec-designer`로 보완이 필요합니다."**

implementer가 BOUNCE_BACK을 보고한 경우도 동일하게 처리한다. 단, question-policy.md를 먼저 적용한다 — 조사로 해소 가능한 모호함이면 해결안을 만들어 처리하고, 진짜 스펙 결함만 되돌린다.

## 구현 공통 지침

- **깊이있게 사고한다**: 엣지케이스와 동시성까지. 스펙의 "이슈 및 결정 사항"을 반드시 반영한다
- **코드베이스 우선**: 기존 컨벤션을 반드시 따른다
- **사이드이펙트 분석**: 변경의 호출 체인을 추적한다
- **테스트는 AC 기준**: 코드 맞춤 테스트는 버그를 통과시킨다
- **스펙 준수**: 스펙에 없는 기능을 추가하지 않는다
- **컨텍스트 경제**: 분석은 code-explorer로, 큰 구현은 implementer로. 메인 세션은 조율·검증·유저 소통에 컨텍스트를 쓴다
