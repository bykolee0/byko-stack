# 구현 루프 프로토콜 (전략 C)

메인 세션이 오케스트레이터로서 AC 단위 구현 루프를 운영한다. ralph-loop(외부 무인 반복)의 대체물 — 세션 안에서 유저와 함께 돌리되, 코드 작성은 implementer 서브에이전트에 위임하여 컨텍스트를 보존한다.

## progress.md

```markdown
# 구현 진행 현황

> 스펙: <경로> / 계획: <경로> / Traceability: <경로>
> 마지막 업데이트: YYYY-MM-DD HH:MM (라운드 N)

## 전체 상태: 🔄 진행중 (k/M AC 완료)

## AC 추적
| AC ID | 담당 | 구현 | 테스트 | 검증 | 변경 파일 |
|-------|------|------|--------|------|----------|
| AC-1 | round 1 | ✅ | ✅ | ✅ | src/... |
| AC-2 | round 2 | 🔄 | ⬜ | ⬜ | |

## 라운드 로그
### Round 1 (YYYY-MM-DD HH:MM)
- 분배: AC-1 (implementer), AC-3 (implementer) — 병렬
- 결과: AC-1 DONE / AC-3 BOUNCE_BACK (사유)

## 이슈 로그
<BOUNCE_BACK 사유, 보류된 [TBD], 발견된 개선거리 (스코프 밖)>
```

## 라운드 절차

### 1. 선택

progress에서 미완료 AC 중 의존성이 풀린 것을 고른다. 서로 다른 파일을 만지는 독립 단위만 병렬 후보로 묶는다 — **같은 파일을 수정하는 단위를 동시에 띄우지 않는다** (충돌).

### 2. 분배 — implementer 호출

각 단위에 전달할 것:

- 해당 AC 전문 (요구사항 + 검증 방법) — "스펙을 읽어라"가 아니라 발췌를 직접 준다
- 스펙 경로 (전체 맥락이 필요할 때 참조용)
- 컨벤션 노트: `analysis/impl-*.md` 경로
- progress.md 경로 (해당 단위 상태 갱신 책임)
- 대상 파일 힌트 (traceability에서)

독립 단위들은 같은 턴에 병렬로 띄운다. 적정 동시 수: 2-3개 (수합·검증이 병목이 되지 않는 선).

### 3. 수합·검증

implementer 보고를 그대로 믿지 않는다:

- 변경 파일 목록과 실제 diff 확인 (`git diff --stat` 수준 — 전문을 다 읽지 않는다)
- 테스트를 직접 실행하여 통과 재확인
- DONE → traceability·progress 갱신
- BLOCKED/BOUNCE_BACK → 이슈 로그 기록, Bounce-Back 규칙 적용

### 4. 방향 점검 (3라운드마다 또는 절반 지점)

- 전체 관련 테스트 실행 (회귀 확인)
- traceability에서 AC 커버리지 확인
- plan과 실제 구현의 괴리 확인 — 있으면 남은 계획 조정, progress에 기록
- 의미 단위로 git commit
- 유저에게 중간 보고 (진행률, 이슈, 다음 라운드 계획)

### 5. 종료 조건

모든 AC가 구현·테스트·검증 완료 → progress를 "✅ 완료"로, 매니페스트 갱신 → SKILL.md Step 5(검증)로.

## 컨텍스트 한계 대응

라운드를 거듭하며 메인 컨텍스트가 한계에 가까워지면:

1. progress.md, traceability.md, 매니페스트를 최신 상태로 만든다
2. 유저에게: "컨텍스트가 한계에 가깝습니다. 진행 상태는 모두 파일에 기록되어 있어, 새 세션에서 `/spec-dev`를 호출하면 이어서 진행됩니다."

이 재개 가능성이 매니페스트+progress 구조의 존재 이유다 — 파일이 상태를 가지므로 세션은 갈아끼울 수 있다.
