# Review: Fresh-Eyes 리뷰

산출물을 **문제 정의에서부터** 다시 본다. eval-gate가 "스펙·코드베이스와 정합한가"를 본다면, review는 "애초의 문제를 올바르고 성실하게 풀었는가"를 본다 — 스펙에 완벽히 부합하는 구현도, 스펙 자체가 문제를 잘못 풀었다면 여기서 걸린다.

먼저 읽을 것: `../../shared/workflow.md`.

직접 리뷰하지 않는다 — 작업 과정을 아는 세션의 리뷰는 결론을 정당화하는 방향으로 기운다. 격리된 `byko-stack:reviewer`가 수행한다.

---

## 사용법

```
/review                     # 매니페스트에서 최신 산출물 자동 선택
/review spec                # 스펙 리뷰
/review implementation      # 구현 리뷰
/review <임의 경로>          # 특정 산출물 리뷰
```

---

## 이 스킬이 트리거되면

### Step 1: 대상과 문제 정의 확보

대상 해석: 인자 > 매니페스트 (workflow.md 탐색 규칙) > 유저 확인.

**문제 정의 확보** — reviewer의 기준점이므로 가장 중요하다:

1. 매니페스트가 있으면 "문제 정의" 섹션을 쓴다
2. 없으면 유저와의 대화에서 원본 요구를 2-5줄로 정리하고 유저에게 확인받는다 — 스펙 요약이 아니라 **스펙 이전의 문제**를 적는다 (스펙을 기준점으로 주면 스펙의 오류를 검출할 수 없다)

### Step 2: reviewer 호출

`byko-stack:reviewer` 서브에이전트에 전달할 것:

- 문제 정의 (Step 1에서 확보한 것)
- 리뷰 대상 경로 — spec 리뷰면 스펙 문서, implementation 리뷰면 변경 코드 경로 + 스펙 경로
- 결과 저장 경로: `<작업 디렉토리>/review-results/<mode>-<timestamp>.md`

**누설 금지**: 현재 세션의 결론, 자체 검증 결과, eval-gate 판정, "잘 된 것 같다" 같은 기대를 전달하지 않는다. reviewer가 백지에서 보는 것이 이 스킬의 전부다.

### Step 3: 결과 검증 + 보고

reviewer의 지적을 그대로 믿지 않는다. 각 지적의 근거(파일:라인, 인용)를 직접 확인하여 accepted / rejected / needs_followup으로 분류한다.

**판정 처리:**

| reviewer 판정 | 의미 | 행동 |
|---------------|------|------|
| **SOUND** | 문제에 맞는 성실한 답 | 매니페스트 갱신, 다음 단계 제안 |
| **CONCERNS** | 방향은 맞으나 실질적 문제 | accepted 항목을 수정 계획으로 정리, 유저와 합의 후 해당 스킬로 |
| **RETHINK** | 문제를 잘못 풀었거나 접근 재고 필요 | 근거를 유저에게 상세히 제시 — 받아들이면 `/spec-designer`로 복귀, 기각하면 사유를 매니페스트에 기록 |

**보고 원칙**: 유저가 다른 파일을 열지 않고 판단할 수 있도록, 각 지적에 내용 설명·실제 인용·문제점·수정 방향을 포함한다. 결과 파일 경로를 알린다.

매니페스트가 있으면 review 결과 상태를 갱신한다.

### Step 4: 다음 단계

- SOUND + 구현 리뷰였음 → 📍 작업 완료. 남은 단계가 있으면 안내
- SOUND + 스펙 리뷰였음 → 📍 `/eval-gate spec` (정합성도 보려면) 또는 `/spec-dev`
- CONCERNS/RETHINK → 수정 담당 스킬 안내 (위 표)

## eval-gate와의 사용 구분

| | eval-gate | review |
|---|-----------|--------|
| 질문 | 상위 산출물·코드베이스와 정합한가 | 애초 문제를 올바르고 성실하게 풀었는가 |
| 기준점 | 스펙/계획/체크리스트 | 매니페스트의 문제 정의 |
| 잡는 것 | AC 누락, 코드 불일치, 깨진 테스트 | 문제 재해석, 컨벤션 위반, 때우기 구현, 과설계 |
| 시점 | 단계 전환 게이트 | 스펙 확정 전, 구현 완료 후, 찜찜할 때 언제든 |

둘 다 통과하면 "맞게 만들었고, 맞는 것을 만들었다"가 된다.
