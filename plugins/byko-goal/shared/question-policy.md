# 질문 정책: 조사 → 제안 → 선택

byko-goal의 설계 단계(goal-design)가 따른다. 목적은 두 실패 모드를 동시에 막는 것:

- **임의 진행**: 모호한 항목을 조용히 자체 결정 → 유저가 원치 않는 방향
- **과잉 질문**: 조사하면 답이 나오는 것까지 질문 → 유저의 시간 낭비

원칙: **유저의 시간은 "정말 유저만 내릴 수 있는 결정"에만 쓴다.** 나머지는 조사하고 결정하고 근거와 함께 기록한다.

이 정책은 도메인 무관이다 — 소프트웨어든 집필이든 기획이든, "조사할 수 있는 사실"과 "유저가 정해야 할 방향"의 구분은 동일하다.

## 절차

모호한 항목을 만나면:

### 1. 먼저 조사한다

도메인에 맞는 출처를 판다:
- 소프트웨어: 코드베이스(직접 Grep/Read 또는 `explorer` 위임)
- 집필/리서치: 기존 자료, 1차 출처, 웹(WebSearch/WebFetch)
- 기획/운영: 기존 문서, 규정, 선례
- 공통: 이미 만든 산출물(goal.md, knowledge.md, 이전 결정)

### 2. 조사로 답이 특정되면 — 질문하지 않는다

채택하고 `from-research`로 기록한다(근거 필수: 파일:라인 / URL / 출처). 유저에게는 "확인한 사실" 목록으로 보고만 한다 — 답변을 기다리는 질문으로 만들지 않는다.

### 3. 답이 하나로 특정되지 않으면 — 해결안을 작성한다

해결안 2-3개를 만든다. 각각에: 근거(기존 패턴과의 정합성), 트레이드오프, 추천 여부와 이유.

### 4. 유저 결정 조건에 해당하는가?

다음 중 하나라도 해당하면 **선택지 + 추천과 함께** 유저에게 묻는다:

| 조건 | 예시 (도메인 무관) |
|------|------|
| 제품/결과물의 요구사항·범위 | 기능 범위, 대상 독자/사용자, 다룰/뺄 주제, 톤 |
| 정책·기준 | 품질 기준선, 사실성 기준, 재시도 정책, 출처 정책 |
| 기존 컨벤션과 충돌 또는 새 컨벤션 정립 | 새 구조/포맷 도입, 용어 체계 |
| 방향성 | 아키텍처, 접근법, 의존성·도구 추가 |
| 되돌리기 어려운 사이드이펙트 | 공개 산출물, 데이터 마이그레이션, 외부 게시 |
| 비용/보안에 실질적 영향 | 유료 외부 서비스, 권한 모델 |

### 5. 해당하지 않으면 — 추천안을 채택하고 진행한다

`assumed`로 기록한다: 채택안 + 근거 + 검토한 대안. 산출물 제시 시점에 "자체 결정한 사항"으로 모아 보여준다 — 유저가 한 번에 훑고 이의만 제기하면 된다.

**assumed 게이트(≤2)**: 자체 결정이 누적되면 게이트에 걸린다. 걸리면 모아서 일괄 확인을 요청한다. 과도한 임의 진행을 구조적으로 차단한다.

## 질문의 형태

- **열린 질문 금지.** "어떻게 할까요?" (X) → 조사 기반 선택지 2-3개 + 추천 (O)
- 한 번에 3개 이하. 각 선택지에 트레이드오프 1줄
- "나중에"라고 하면 `[TBD]`로 기록하고 진행 — 해당 task 실행 시점에 다시 결정

## 자기 점검

- 이 질문의 답이 자료/코드/웹 어딘가에 있는가? → 있으면 묻지 말고 찾는다
- 해결안 없이 열린 질문을 던지려 하는가? → 먼저 조사해 선택지를 만든다
- 유저 결정 조건 표에 없는데 묻고 있는가? → 추천안 채택 + assumed 기록으로 전환
- 유저 결정 조건에 해당하는데 조용히 진행하려 하는가? → 멈추고 묻는다
