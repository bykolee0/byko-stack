# Task 분해 — 크기 기준과 패턴

마스터 체크리스트의 **항목 하나 = 한 세션(worker)의 목표**다. 분해가 자율 루프의 성패를 가른다. 잘못 쪼개면: 너무 크면 한 worker 컨텍스트에 안 들어와 중간에 터지고, 너무 잘면 세션 시작·검증 오버헤드가 본작업을 압도한다.

## 한 task의 크기 기준

좋은 task는 다음을 **모두** 만족한다:

1. **단일 컨텍스트 완결성** — 빈 컨텍스트로 시작한 worker가 (goal.md + knowledge.md + 최신 handoff를 읽고) 끝까지 갈 수 있다. 산출물 분량이 한 세션에 무리면 분해한다.
2. **독립 검증 가능성** — "끝났다"가 객관적으로 판정된다. 완료조건을 검증 방법과 함께 적을 수 있어야 한다. 적을 수 없으면 task가 모호한 것 — 더 구체화하거나 쪼갠다.
3. **자기완결적 인계** — 끝났을 때 다음 task가 이어받을 수 있는 명확한 산출물을 남긴다.
4. **명확한 의존성** — 다른 task의 산출물이 선행돼야 하면 `(의존: NN)`으로 표시한다. 의존이 거미줄이면 경계가 잘못된 것.

## 분해 휴리스틱

- **수직 슬라이스 우선** — "끝까지 동작하는 얇은 조각"으로 쪼갠다. 수평(전부 설계 → 전부 구현 → 전부 테스트)은 중간 산출물이 검증 불가라 자율 루프에 나쁘다.
- **검증 경계로 자른다** — "여기까지면 독립적으로 맞는지 볼 수 있다"가 자연스러운 절단면.
- **5±2 규칙** — 한 목표의 task가 너무 많으면(>~15) 상위 묶음으로 재편하거나 별도 목표로 분리. 너무 적으면(1-2개) 애초에 byko-goal이 과한 것 — byko-stack 단발 워크플로우가 맞다.
- **첫 task는 "기반/정찰"로** — 환경 셋업, 핵심 조사, 스켈레톤. 이게 knowledge.md를 살찌워 이후 task를 빠르게 한다.

## 도메인별 패턴

분해 모양과 검증 방법은 도메인에 따라 다르다. 검증 방법은 **task의 완료조건에 데이터로** 적힌다(에이전트가 아니라).

### 소프트웨어
- task 예: "결제 웹훅 핸들러 구현", "환불 API + 멱등성", "관리자 정산 화면"
- 절단면: 기능 수직 슬라이스(엔드포인트 1개 + 테스트), 또는 AC 묶음
- 검증: 테스트 실행, `git diff` 실측, 빌드/린트
- knowledge 시드: 아키텍처 / 컨벤션 / 함정 / 구조 맵

### 집필 (책·교재·문서)
- task 예: "3장 분수의 덧셈 집필", "용어집 정리", "2부 전체 교차참조 점검"
- 절단면: 챕터/섹션 단위. 리서치가 무거우면 "N장 리서치"와 "N장 집필"을 분리
- 검증: 학습목표/목차 대조, 가독성 기준, 이전 챕터와 모순 확인, 핵심 사실 웹 교차확인
- knowledge 시드: 확정 사실·출처 / 용어·표기 / 톤·문체 / 챕터 의존 / 미해결 리서치

### 기획·운영 (프로젝트/이벤트/비즈니스 구성)
- task 예: "환불 정책 문서 확정", "벤더 비교·선정", "런북 작성"
- 절단면: 산출 문서/결정 단위
- 검증: 규정·요구사항 체크리스트 대조, 기존 문서와 정합, 필수 항목 포함 여부
- knowledge 시드: 확정 결정·근거 / 제약·규정 / 이해관계자 / 미결 항목

### 리서치
- task 예: "경쟁사 가격 모델 조사", "출처 신뢰도 검증", "최종 보고서 합성"
- 절단면: 하위 질문 단위, 마지막에 합성 task
- 검증: 주장마다 1차 출처, 교차 확인, 반대 증거 탐색
- knowledge 시드: 확인된 사실(출처) / 상충하는 증거 / 미확인 / 방법론

## 안티패턴

- ❌ "전체 구현" 같은 한 덩어리 task — 한 세션에 안 끝난다
- ❌ 검증 방법을 적을 수 없는 task — 완료를 판정할 수 없다
- ❌ "리팩터링/개선" 류 무한정 task — 종료 조건이 없다. 구체적 산출물로 한정한다
- ❌ 모든 task가 서로 의존 — 경계가 틀렸다. 독립 슬라이스로 다시 자른다
