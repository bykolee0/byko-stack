# Goal Design — 자율 실행 목표 설계

유저와의 **대화**로 장기 목표를 설계하고, 이후 자율 루프(`goal-run`)가 읽고 실행할 **문서 세트**를 만든다. 이 문서 세트가 모든 후속 세션의 유일한 지침이 되므로, 여기서의 품질이 전체를 좌우한다.

먼저 읽을 것: `../../shared/question-policy.md`(질문 정책), `../../shared/templates.md`(문서 형태), `../../shared/loop-protocol.md`(이후 어떻게 실행되는지 — 이걸 알아야 task를 올바른 크기로 쪼갠다).

핵심 원칙:
- **모호한 상태를 통과시키지 않는다** — 단, 조사로 답할 수 있는 것은 유저에게 묻지 않는다 (question-policy).
- **도메인을 먼저 못 박는다** — 검증 기준은 목표에 따라 달라진다. 결제사이트면 기술스택을, 교육책자면 대상독자·수준·사실성 기준을 캐물어야 한다.
- **메인 세션은 대화와 조율만** — 넓은 조사는 `byko-goal:explorer`에 위임하고 결론만 컨텍스트에 유지한다.

---

## 이 스킬이 트리거되면

### Step 0: 기존 목표 확인
`docs/goals/*/`를 탐색한다. 진행 중인 goal이 이 요청과 관련 있으면 현황을 요약하고 이어갈지(설계 보완) 또는 바로 `/goal-run`으로 실행할지 안내한다. 새 목표면 진행한다 — 이 시점에 파일은 만들지 않는다.

### Step 1: 첫 응답 — 도메인·목표·검증 접근을 잡는다
유저의 요청에서 **도메인**을 먼저 파악한다(코드 / 집필 / 기획 / 리서치 / 기타). 도메인이 검증 방식과 질문 항목을 결정한다.

코드베이스나 자료가 있으면 `explorer`에 초기 조사를 위임한다(범위가 좁으면 직접). 그 위에서 첫 응답을 구성한다:
- **확인한 사실** — 조사로 알아낸 것 (근거: 파일:라인 / URL) — 보고만, 질문 아님
- **갈리는 지점** — 해결안 2-3개 + 추천 + 트레이드오프
- **유저만 정할 수 있는 것** — question-policy §4 조건 해당분만

특히 다음을 향해 대화를 끈다:
- **목표**와 완성됐을 때의 상태
- **완수 조건**: 공통(전 체크리스트 완료)은 고정. **최종 조건(옵션)** — 결과/동작 기준과 그 **검증 방법**(테스트 명령 / 루브릭 / 산출물 정합 등). 도메인별로 형태가 다르다.
- **비범위(non-goals)** — 무엇을 안 할지

⏸️ **유저 답변을 기다린다.**

### Step 2~N: 깊이 파기 — 충분해질 때까지
"결제사이트 만들어줘"처럼 넓으면, 한 번에 템플릿을 채우려 하지 말고 **결정할 게 많은 지점을 계속 캐묻는다**(기술/인프라/서드파티/정책…). 매 라운드: 답변 반영 → 필요 시 추가 조사(explorer) → 남은 모호 항목을 question-policy대로 처리(`from-research`/`assumed`/질문).

**충분 기준** — 아래가 대부분 특정될 때까지(유저 확인 또는 조사 확인):

> 목표 · 범위/비범위 · 완수 조건과 검증 방법 · 도메인 컨텍스트와 제약 · 대상(독자/사용자) · 컨벤션(톤·스타일·기술스택) · 핵심 결정 · **세션 크기로 분해된 task 목록**

기준: blocking 0, assumed ≤2, open ≤3. assumed가 게이트를 넘으면 모아서 일괄 확인한다.

### Step 3: task 분해 — 가장 중요한 단계
마스터 체크리스트의 **각 항목 하나가 한 세션의 목표**다. 크기 기준과 분해 패턴은 `references/task-sizing.md`를 읽고 따른다. 요점: **한 worker의 컨텍스트에 들어오고 + 독립 검증 가능한 단위**로. 너무 크면 분해, 너무 잘면 병합. 의존성이 있으면 표시한다.

각 항목에 **수용 힌트**(끝나면 무엇이 참이어야)를 단다 — 이게 나중에 worker가 tasks/NN.md 완료조건을 세우는 출발점이다.

### Step 4: 캡 입력받기 (유저 결정)
자율 루프는 fire-and-forget이라 **방어막**이 필수다. 유저에게 묻는다(추천값 제시):
- `max_iterations` — worker dispatch(≈ task 시도) 횟수 상한 (추천: task 수 × 2 정도)
- `max_minutes` — 벽시계 상한

이건 유저만 정할 수 있다(비용/시간 통제). 짧게 묻고 추천값을 제시한다.

회귀·재계획 노브는 기본값으로 두되 유저가 원하면 조정한다 — `checkpoint_every`(기본 4 task: auditor가 회귀·드리프트를 점검하는 주기), `major_changes_budget`(기본 3: auditor의 파괴적 체크리스트 변경 허용 횟수). 둘 다 run-state.md에 기록한다.

### Step 5: 문서 세트 생성
`docs/goals/<slug>/`에 생성한다(`templates.md` 따름). `<slug>`는 목표를 요약한 kebab-case:
1. **goal.md** — 목표, 완수조건(공통+최종), 공통 컨텍스트, 마스터 체크리스트(수용 힌트 포함), 인덱스
2. **run-state.md** — 캡·노브(Step 4의 max_*, checkpoint_every, major_changes_budget), 카운터 0, task 상태표, 빈 산출물 소유 맵
3. **knowledge.md** — 도메인에 맞게 시드한 섹션(코드면 아키텍처/컨벤션/함정, 집필이면 확정사실/출처/용어/톤 등). 내용은 비워두되 골격을 깔아 첫 worker가 채우기 쉽게.
4. `tasks/`, `handoffs/`, `eval/`, `audit/` 빈 디렉토리

유저에게 보여줄 때: 마스터 체크리스트 + 완수조건 + **"자체 결정한 사항(assumed)"** 목록 — 한 번에 훑고 이의만 제기하도록.

⏸️ **유저 피드백을 기다린다. 만족할 때까지 반복.**

### Step 6: 다음 단계로 인계 (필수 — 여기서 끝내지 않는다)
설계는 시작일 뿐이다. **반드시 실행 단계로 이어지게 안내한다.**

1. 권장 권한 설정을 안내한다 — 자율 루프는 무프롬프트가 필요하다. `.claude/settings.local.json`에 자동승인을 넣어줄지 제안하고, 동의하면 작성한다:
   ```json
   { "permissions": { "defaultMode": "bypassPermissions" } }
   ```
   (더 안전하게 가려면 `acceptEdits` — 편집은 자동, 위험한 bash만 확인. fire-and-forget엔 `bypassPermissions` 추천.)

2. **실행 커맨드를 그대로 출력한다:**
   > 📍 **설계 완료. 자율 실행을 시작하려면:**
   > - 무프롬프트 실행(권장): 새 창에서 `claude --dangerously-skip-permissions` → `/goal-run <slug>`
   > - 또는 settings를 넣었다면 재시작 후 `/goal-run <slug>`
   > - 지금 이 세션에서 바로 시작도 가능: `/goal-run <slug>` (권한 프롬프트가 뜰 수 있음)

3. 유저가 "바로 시작해"라고 하면 `Skill`로 `goal-run`을 호출한다(또는 위 커맨드 안내). 그렇지 않으면 위 안내로 마친다.

## 중단 후 재개
goal.md가 이미 있으면 그것으로 현황을 복원한다: 체크리스트 진행 상태 요약 → 설계 보완이 필요한지, 아니면 바로 `/goal-run`인지 안내.

## 에이전트 행동 지침
1. **깊이 사고한다** — 목표를 이루기 위한 결정·엣지·실패 시나리오까지 공통 컨텍스트와 task에 반영한다.
2. **첫 응답부터 조사한다** — 조사로 답할 수 있는 걸 유저에게 묻는 것은 이 스킬의 실패다.
3. **도메인을 못 박고 검증 방법을 데이터로 남긴다** — 검증은 에이전트가 아니라 문서가 정의한다.
4. **task 분해에 가장 공들인다** — 잘못 쪼개면 자율 루프 전체가 흔들린다.
5. **반드시 다음 단계로 인계한다** — 설계만 하고 멈추지 않는다.
